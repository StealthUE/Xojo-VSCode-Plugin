### How to do common tasks

#### Find a method's source
Read the `.xojo` file from the exports folder for that block/method:
```
exports/{ProjectName}/{BlockType}_{BlockName}/{MethodName}.xojo
```
Line 2 of every file is the full method signature as a comment.

#### Search for a pattern across all code
Grep **the exports folder**, not the XML file:
```bash
grep -rn "pattern" "exports/{ProjectName}/"
```
Example — find all ByRef parameters of a certain type:
```bash
grep -rn "ByRef ResultArr" "exports/{ProjectName}/"
```

#### Find which methods call a given method
Use the VS Code command **Xojo: Find Callers** (right-click a method in the Xojo Explorer
tree). This searches the exports folder and writes results to `_callers.json` in the edits
directory so you can read them programmatically.

Alternatively grep the exports folder directly:
```bash
grep -rn "MethodName(" "exports/{ProjectName}/"
```

#### Understand the project structure
Read `CODEBASE.md` in the exports root. It lists every block with all its methods,
events, and properties, and shows which `.xojo` file each one maps to.

#### Find overloads of a method
Read `_overloads.json` in the block's export subfolder. It lists every overload with its
full signature and filename.

#### Edit a method
1. Write to the `.xojo` file in the exports folder using your normal file-write tool
2. Leave line 1 (the `// vsxojo:` metadata header) completely untouched
3. **Do not touch the `.xojo_xml_project` file** — VSXojo writes the change back to XML
   automatically within ~300 ms of detecting your file write

**Claude Code workflow:** Write the file with the Edit or Write tool. VSXojo detects the
change via a filesystem watcher and writes it back to XML immediately — no Ctrl+S or VS Code
interaction required.

**Do NOT edit the XML file directly** to fix method body content. The `.xojo` file is the
source of truth; the XML is the output. Direct XML edits will be overwritten on next export.

### Creating new items (modules, classes, methods, properties)

Use the **creation-request protocol** — write a JSON file, wait for the result, then edit
the generated export files.

**Step 1 — Write a request file** to any location inside the globalStorage folder.  
Suggested path: `{globalStoragePath}/_xojo_create.json`  
The exact globalStoragePath is shown in CODEBASE.md.

**Step 2 — Read the result file** at the same path but named `_xojo_create_result.json`.
The extension deletes the request file and writes the result within ~1 second.

**Step 3 — Edit the new export files** (they appear in the exports folder after creation).

#### Request file formats

**New top-level module:**
```json
{ "action": "newModule", "name": "MyModule" }
```

**New top-level class:**
```json
{ "action": "newClass", "name": "MyClass", "superclass": "BaseClass" }
```
`superclass` is optional — omit for no superclass.

**New method on an existing class or module:**
```json
{ "action": "newMethod", "blockName": "MyClass", "name": "DoSomething", "params": "x As Integer, y As String", "returnType": "Boolean" }
```
- `blockName` — case-insensitive name of the class/module to add to (check CODEBASE.md)
- `params` — parameter list string; omit or `""` for no parameters
- `returnType` — return type for Functions; omit or `""` for Subs (void)

**New property on an existing class or module:**
```json
{ "action": "newProperty", "blockName": "MyClass", "name": "MyProp", "type": "String", "defaultValue": "\"\"" }
```
- `type` — required; e.g. `"String"`, `"Integer"`, `"Boolean"`
- `defaultValue` — optional

**Add an event handler to an existing class or module:**
```json
{ "action": "newEvent", "blockName": "MyClass", "name": "Open", "params": "", "returnType": "" }
```
Same fields as `newMethod` — omit `returnType` for Sub event handlers.

**Add a constant to an existing class or module:**
```json
{ "action": "newConstant", "blockName": "MyClass", "name": "MAX_SIZE", "value": "100" }
```
- `value` — the constant's value as a string
- `isString` — optional boolean; `true` forces string (hex) encoding. If omitted, auto-detected:
  pure numbers and `true`/`false` use `<ItemValue>`; anything else is hex-encoded as a string constant.

#### Result file format
```json
{ "success": true, "id": "...", "message": "Class \"MyClass\" created" }
```
or on error:
```json
{ "success": false, "error": "Block \"MyClass\" not found. Available: ..." }
```

**After a successful creation**, the exports folder is automatically updated — the new
block's export subfolder and method/property files will already exist when you go to edit
them.

#### Check that your edit was saved to XML
Use the VS Code command **Xojo: Check Sync Status**. Results are written to `_sync.json`
in the edits directory.

## What is a Xojo project file?

A `.xojo_xml_project` (or `.xojo_xml_code`) file is a large XML file.
The root element contains a **flat list of `<block>` elements** — every class, module,
window, folder, and external code reference is a `<block>`. The hierarchy is NOT
expressed through XML nesting; instead it is encoded via a `containerId` field.

---

## Block structure

```xml
<block type="Module" ID="12345">
  <ObjName>MyClass</ObjName>
  <ObjContainerID>0</ObjContainerID>   <!-- 0 = top-level; otherwise = parent Folder ID -->
  <IsClass>1</IsClass>
  <Superclass>SomeBase</Superclass>

  <Property>
    <ItemName>MyProp</ItemName>
    <ItemDeclaration>MyProp As String = "hello"</ItemDeclaration>
    <PartID>42</PartID>
  </Property>

  <Constant>
    <ItemName>MAX_SIZE</ItemName>
    <ItemValue>100</ItemValue>
  </Constant>

  <Method>
    <ItemName>DoSomething</ItemName>
    <ItemParams>x As Integer</ItemParams>
    <ItemResult>String</ItemResult>       <!-- return type; absent for Subs -->
    <PartID>77</PartID>
    <ItemSource>
      <SourceLine>Function DoSomething(x As Integer) As String</SourceLine>
      <SourceLine>  Return Str(x)</SourceLine>
      <SourceLine>End Function</SourceLine>
    </ItemSource>
  </Method>

  <HookInstance>                         <!-- event handler -->
    <ItemName>Open</ItemName>
    <PartID>88</PartID>
    <ItemSource>
      <SourceLine>Sub Open()</SourceLine>
      <SourceLine>  // init code</SourceLine>
      <SourceLine>End Sub</SourceLine>
    </ItemSource>
  </HookInstance>

  <Note>
    <ItemName>Design Notes</ItemName>
    <ItemSource>
      <SourceLine>This class handles...</SourceLine>
    </ItemSource>
  </Note>
</block>
```

---

## Building the tree (containerId hierarchy)

Blocks are **flat** in the XML. To reconstruct the parent-child tree:

- `ObjContainerID = "0"` → top-level block
- `ObjContainerID = "<some ID>"` → child of the block whose `ID` attribute matches

A `Folder` block (type `"Folder"`) is a pure organisational container with no code of its own.
All blocks whose `containerId` equals the folder's `ID` are its children.

---

## Block types

| `type` attribute | Meaning |
|---|---|
| `Module` | Module or Class — check `<IsClass>` (1 = class) |
| `Folder` | Organisational folder; no code, just groups children |
| `ExternalCode` | Reference to a separate `.xojo_xml_code` file |
| `WebView` | Web page / window |
| `WebContainer` | Reusable web UI container |
| `WebSession` | Session class |
| `Project` / `ProjectSettings` / `UIState` | Internal Xojo metadata — no user code |

---

## Code items inside a block

| XML element | What it is | Key child elements |
|---|---|---|
| `<Method>` | Sub or Function | `<ItemName>`, `<ItemParams>`, `<ItemResult>` (return type), `<ItemSource><SourceLine>`, `<PartID>` |
| `<HookInstance>` | Event handler | Same structure as Method |
| `<Property>` | Instance variable | `<ItemName>`, `<ItemDeclaration>` (`"name As Type = default"`), `<PartID>` |
| `<Constant>` | Named constant | `<ItemName>`, `<ItemValue>` or `<ItemDef><Hex>` (hex-encoded string value) |
| `<Note>` | Developer note / comment block | `<ItemName>`, `<ItemSource><SourceLine>` |

**`<SourceLine>`** elements inside `<ItemSource>` form the lines of code (or text).
There may be one or many — always treat them as an array.

---

## ExternalCode blocks

An `ExternalCode` block points to a separate `.xojo_xml_code` file.

```xml
<block type="ExternalCode" ID="222">
  <ObjName>LibFoo</ObjName>
  <PartialPath>..\Lib\Foo.xojo_xml_code</PartialPath>
  <FullPath>C:\absolute\path\Foo.xojo_xml_code</FullPath>
</block>
```

Resolve the path: use `FullPath` if it exists on disk; otherwise resolve `PartialPath`
relative to the directory containing the main project file.
The referenced `.xojo_xml_code` file has the same flat-block XML format.

## Exported file format (VSXojo temp files)

The VSXojo extension deconstructs the project into individual `.xojo` files in a
temp folder (`<VS Code global storage>/exports/<projectName>/`).

Each method/event file looks like this:

```
// vsxojo:sourceFile="C:\...\MyProject.xojo_xml_project"|partId="77"|xmlTag="Method"|itemName="DoSomething"|sig="Function DoSomething(x As Integer) As String"|isFunction=true
// Function DoSomething(x As Integer) As String

  Return Str(x)
```

- **Line 1** — machine-readable metadata (do not modify); used to write changes back to the XML
- **Line 2** — human-readable signature comment
- **Remaining lines** — the method body only (no Sub/Function header or End Sub/End Function footer)

A `_properties.xojo` file in each block folder lists all properties in declaration format:
```
MyProp As String = "hello"
AnotherProp As Integer
```

A `_manifest.json` file lists all blocks, their methods, events, and properties in structured form.

`CODEBASE.md` at the export root is an AI-readable summary of the entire project.

## Xojo language basics

> **Full language reference:** See `XOJO_HELP.md` in this directory for complete Xojo language
> docs — data types, OOP, API 2.0 class renames, error handling, threading, file I/O,
> databases, networking, advanced features, and compiler directives.

- Case-insensitive keywords: `Sub`, `Function`, `End Sub`, `End Function`, `Dim`, `If`, `Then`, `Else`, `For`, `Next`, `While`, `Wend`, `Select`, `Case`, `Return`
- Types: `Integer`, `Double`, `String`, `Boolean`, `Variant`, `Auto`, plus class names
- String concatenation: `+` or `&`
- Nil check: `If obj = Nil Then`
- Arrays: `Dim arr() As String` / `arr.Append(...)` / `arr.Count`
- Comments: `//` or `'`
- No semicolons; one statement per line
- `Me` = current instance (equivalent to `this`)
- `Self` = same as `Me` in most contexts
